# CareerPilot Agent 开发计划

更新日期：2026-07-23

## 1. 项目目标

CareerPilot Agent 是一个面向 AI / AI Agent 实习求职场景的智能助手。用户提供岗位 JD、个人简历和项目资料后，系统将逐步实现岗位要求解析、能力匹配、差距分析、简历建议和学习计划生成。

项目强调三个原则：

- **有依据**：回答尽量引用 JD、简历或知识库中的原始资料。
- **可验证**：核心函数有自动化测试，重要功能有明确验收标准。
- **保护隐私**：真实简历只在本地处理，公开仓库只保存脱敏样例。

## 2. 当前最小 RAG 架构

```text
用户输入文档路径和问题
        ↓
文档读取：TXT / Markdown / PDF
        ↓
固定字符长度切分
        ↓
英文关键词重合检索
        ↓
构造带 Context 的 Prompt
        ↓
调用 DeepSeek 生成回答
        ↓
显示答案、来源文件和检索片段编号
```

## 4. DOCX 文档解析增强

状态：已完成

### 4.1 用户问题

真实用户的简历通常是 DOCX 或 PDF。产品不应该要求用户先手动转换成 Markdown，因此文档读取层需要负责格式解析，并向后续切分和检索流程提供统一文本。

### 4.2 技术方案

- TXT / Markdown：继续使用 `pathlib.Path.read_text()`。
- PDF：继续使用 PyMuPDF。
- DOCX：使用 Microsoft MarkItDown 在本地转换成 Markdown 文本。
- 转换结果默认只保存在内存中，不额外生成 `.md` 文件。
- 所有格式最终统一返回字符串，再交给 `split_text()`。

选择 MarkItDown 的原因：

- 免费、开源，采用 MIT 许可证。
- 支持本地处理 DOCX，不需要上传用户文件。
- 相比只读取段落的简单方案，更适合保留标题、列表和表格等结构。
- 当前项目只安装 DOCX 所需的可选依赖，避免引入暂时用不到的全部格式依赖。

### 4.3 开发步骤

1. 安装 MarkItDown 的 DOCX 依赖并更新 `requirements.txt`。
2. 先添加一个 DOCX 读取失败测试。
3. 在 `rag_ingest.py` 中增加 DOCX 分支。
4. 运行针对性测试并转为绿灯。
5. 运行完整离线测试，检查已有 TXT、Markdown、PDF 和 RAG 行为。
6. 使用本地真实简历做手动验证，但不提交原始简历。

完成证据：

- 自动化测试共 8 个，全部通过。
- 使用本地 DOCX 简历验证，成功提取非空文本。
- 验证过程只输出字符数，没有展示或提交简历正文。

### 4.4 验收标准

- `read_text_file()` 接受 `.docx` 文件路径。
- DOCX 解析结果是非空字符串。
- 解析结果包含简历中的可见正文内容。
- 原有 TXT、Markdown 和 PDF 行为不受影响。
- 不调用外部文档解析 API。
- 自动化测试不依赖真实个人简历。
- README 和依赖清单与实际实现保持一致。

### 4.5 隐私约束

- 真实简历不得提交到公开 GitHub 仓库。
- 自动格式转换不等于脱敏。
- 公开测试数据必须删除姓名、电话、邮箱、详细地址和其他个人标识。
- 后续上传功能应明确说明文件用途、存储位置和删除策略。

## 5. 后续迭代方向

1. 用 Embedding 和向量数据库替换英文关键词检索。
2. 为切分片段保存页码、段落号和文档类型等元数据。
3. 输入 JD 和简历，生成结构化能力匹配报告。
4. 使用 Agent 工作流拆分 JD 解析、简历解析和差距分析。
5. 增加评估、FastAPI、Web UI、Docker 和部署能力。

## 6. Git 工作方式

- `main` 只保存已经验证并合并的功能。
- 每个功能使用独立的 `feature/...` 分支。
- 功能开发遵循“红灯测试 → 最小实现 → 全量验证”。
- 提交前检查测试、暂存文件范围和 `git diff --cached --check`。
- 通过 Pull Request 合并，并在合并后重新测试 `main`。
